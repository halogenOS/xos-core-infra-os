# Stalwart + TMail: Docker Setup Guide (Arch Linux)

This guide reproduces the halogenOS mail stack (Stalwart Mail Server +
TMail webmail) inside Docker on an Arch Linux host with a rooted Docker
daemon. It mirrors the NixOS configuration in this repository.

## Architecture overview

```
                        ┌─────────────────────────────────────────┐
                        │              Docker host                │
  Internet              │                                         │
  ───────►  :25  ───────┤──► stalwart  (SMTP inbound)            │
  ───────►  :465 ───────┤──► stalwart  (SMTP submission, TLS)    │
  ───────►  :993 ───────┤──► stalwart  (IMAPS)                   │
  ───────►  :443 ───────┤──► caddy ──► stalwart :8080 (JMAP/web) │
                        │         └──► tmail    :80   (webmail)   │
  ───────►  :80  ───────┤──► caddy     (ACME challenges)         │
                        └─────────────────────────────────────────┘
```

**Stalwart** handles SMTP (25, 465), IMAP (993), and exposes its
management UI + JMAP API on an internal HTTP port. **Caddy** terminates
TLS for HTTPS and routes traffic to Stalwart or TMail based on the
hostname. **TMail** provides the webmail frontend.

## Prerequisites

### 1. Install Docker

```sh
sudo pacman -S docker docker-compose
sudo systemctl enable --now docker.service
```

Verify:

```sh
docker info
docker compose version   # needs v2+
```

### 2. DNS records

All DNS records must be in place *before* starting the stack — ACME
certificate issuance and mail delivery depend on them.

Use the `setup-dns.ts` script in this repository to generate, check,
and optionally apply the full record set:

```sh
# Print the records you need
./scripts/setup-dns.ts print

# Check which ones resolve correctly
./scripts/setup-dns.ts check

# Apply via Cloudflare (if applicable)
./scripts/setup-dns.ts apply
```

At minimum you need:

| Type   | Name                          | Value                                         |
|--------|-------------------------------|-----------------------------------------------|
| A      | `mail.example.com`            | your server IPv4                              |
| AAAA   | `mail.example.com`            | your server IPv6 (if available)               |
| A      | `webmail.example.com`         | your server IPv4                              |
| AAAA   | `webmail.example.com`         | your server IPv6 (if available)               |
| MX     | `example.com`                 | `mail.example.com` (priority 10)              |
| TXT    | `example.com`                 | `v=spf1 mx ra=postmaster -all`               |
| TXT    | `_dmarc.example.com`          | `v=DMARC1; p=reject; rua=mailto:postmaster@…` |
| TXT    | `_smtp._tls.example.com`      | `v=TLSRPTv1; rua=mailto:postmaster@…`        |
| TXT    | `_mta-sts.example.com`        | `v=STSv1; id=1`                               |
| A/AAAA | `mta-sts.example.com`         | your server IPv4/IPv6                         |
| A/AAAA | `autoconfig.example.com`      | your server IPv4/IPv6                         |
| A/AAAA | `autodiscover.example.com`    | your server IPv4/IPv6                         |
| SRV    | `_imaps._tcp.example.com`     | `0 1 993 mail.example.com`                    |
| SRV    | `_submissions._tcp.example.com`| `0 1 465 mail.example.com`                   |
| SRV    | `_autodiscover._tcp.example.com`| `0 1 443 autodiscover.example.com`           |
| SRV    | `_carddavs._tcp.example.com`  | `0 1 443 mail.example.com`                   |
| SRV    | `_caldavs._tcp.example.com`   | `0 1 443 mail.example.com`                   |
| PTR    | your IPv4                     | `mail.example.com` (set at hosting provider)  |

DKIM records (`ed._domainkey` and `rsa._domainkey`) are generated by
Stalwart after first boot — add them once the server is running (see
step 6 below).

### 3. Firewall

Open the required ports on the host:

```sh
# If using iptables
sudo iptables -A INPUT -p tcp --dport 25 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 465 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 993 -j ACCEPT

# If using nftables (recommended on Arch)
sudo nft add rule inet filter input tcp dport { 25, 80, 443, 465, 993 } accept
```

Ensure your hosting provider's firewall (Hetzner, etc.) also allows
these ports.

### 4. Reverse DNS (PTR)

Set the PTR record for your server's primary IPv4 (and IPv6 if
available) to `mail.example.com` via your hosting provider's control
panel. This is critical for mail deliverability — many receiving
servers reject mail from IPs without matching PTR records.

## Directory layout

Create the working directory for the mail stack:

```sh
sudo mkdir -p /opt/mail/{stalwart-data,tmail-data,caddy-data,caddy-config}
cd /opt/mail
```

```
/opt/mail/
├── docker-compose.yml
├── Caddyfile
├── stalwart-data/       # Stalwart persistent storage (mail, db, certs, DKIM keys)
├── tmail-data/          # TMail configuration
├── caddy-data/          # Caddy TLS certificates (ACME)
└── caddy-config/        # Caddy runtime config
```

## Configuration files

### docker-compose.yml

See `scripts/docker-compose.mail.yml` in this repository, or create it
manually — the reference compose file is provided alongside this guide.

Copy it into `/opt/mail/`:

```sh
cp docker-compose.mail.yml /opt/mail/docker-compose.yml
```

Edit the file and replace all placeholder values:

| Placeholder              | Replace with                        |
|--------------------------|-------------------------------------|
| `mail.example.com`       | your mail domain                    |
| `webmail.example.com`    | your webmail domain                 |
| `example.com`            | your org domain                     |
| `postmaster@example.com` | your postmaster address             |

### Caddyfile

Create `/opt/mail/Caddyfile`:

```caddyfile
{
	admin off
}

mail.example.com {
	reverse_proxy stalwart:8080
}

mta-sts.example.com {
	header Content-Type text/plain
	respond `version: STSv1
mode: enforce
mx: mail.example.com
max_age: 604800
` 200
}

autoconfig.example.com {
	reverse_proxy stalwart:8080
}

autodiscover.example.com {
	reverse_proxy stalwart:8080
}

webmail.example.com {
	reverse_proxy tmail:80
}
```

Replace `example.com` with your actual domain throughout. Caddy
automatically obtains and renews TLS certificates via Let's Encrypt.

## Step-by-step deployment

### Step 1: Create the compose file and Caddyfile

```sh
sudo mkdir -p /opt/mail
cd /opt/mail

# Copy or create docker-compose.yml and Caddyfile as described above
sudo cp /path/to/docker-compose.mail.yml docker-compose.yml
# Edit to replace placeholders
sudo vim docker-compose.yml
sudo vim Caddyfile
```

### Step 2: Start the stack

```sh
cd /opt/mail
sudo docker compose up -d
```

Watch the logs to confirm everything starts cleanly:

```sh
sudo docker compose logs -f
```

You should see:
- Caddy obtaining TLS certificates for your domains
- Stalwart starting up and initializing its database
- TMail starting its webmail interface

### Step 3: Initial Stalwart admin setup

On first boot, Stalwart creates a default admin account. Check the
logs for the auto-generated credentials:

```sh
sudo docker compose logs stalwart | grep -i "admin\|password\|credential"
```

If no credentials are printed, the default is typically
`admin` / `changeme`. Access the Stalwart admin UI at
`https://mail.example.com/login` and change the password immediately.

### Step 4: Create your domain

In the Stalwart admin UI (`https://mail.example.com`):

1. Go to **Management** → **Directory** → **Domains**
2. Add your domain (`example.com`)
3. Go to **Management** → **Directory** → **Accounts**
4. Create your first email account (e.g., `postmaster@example.com`)

### Step 5: Configure DKIM signing

Stalwart generates DKIM keys automatically for each domain. To retrieve
them:

**Option A: Via the admin UI**

Navigate to **Management** → **Directory** → **Domains** → your domain.
The DKIM public keys (Ed25519 and RSA) are displayed there.

**Option B: Via the API**

```sh
curl -s -H "Authorization: Bearer YOUR_TOKEN" \
  https://mail.example.com/api/dns/records/example.com | jq .
```

**Option C: Via setup-dns.ts**

Configure your Stalwart API access in `dns.toml`:

```toml
[stalwart]
url = "https://mail.example.com"
token = "your-api-token"
```

Then run:

```sh
./scripts/setup-dns.ts print
```

This fetches the DKIM keys from Stalwart and includes them in the
output.

### Step 6: Publish DKIM records

Add the two DKIM TXT records to your DNS:

| Name                        | Value                           |
|-----------------------------|---------------------------------|
| `ed._domainkey.example.com` | `v=DKIM1; k=ed25519; p=<key>` |
| `rsa._domainkey.example.com`| `v=DKIM1; k=rsa; p=<key>`     |

Then verify:

```sh
./scripts/setup-dns.ts check
```

### Step 7: Test mail delivery

Send a test email to an external address and check:

```sh
# From the host, using openssl to connect to SMTP submission
openssl s_client -connect mail.example.com:465 -quiet
```

Or use any email client configured with:

| Setting         | Value                 |
|-----------------|-----------------------|
| IMAP server     | `mail.example.com`    |
| IMAP port       | 993 (SSL/TLS)         |
| SMTP server     | `mail.example.com`    |
| SMTP port       | 465 (SSL/TLS)         |
| Username        | `user@example.com`    |

Useful external tools for verifying your setup:
- [mail-tester.com](https://www.mail-tester.com) — send a test email and get a deliverability score
- [mxtoolbox.com](https://mxtoolbox.com) — check MX, SPF, DKIM, DMARC
- [internet.nl](https://internet.nl) — comprehensive email standards test

### Step 8: Access webmail

Open `https://webmail.example.com` in your browser. Log in with your
email credentials.

## Operations

### Viewing logs

```sh
# All services
sudo docker compose logs -f

# Single service
sudo docker compose logs -f stalwart
sudo docker compose logs -f caddy
sudo docker compose logs -f tmail
```

### Restarting services

```sh
sudo docker compose restart stalwart
sudo docker compose restart caddy
sudo docker compose restart tmail
```

### Updating images

```sh
cd /opt/mail
sudo docker compose pull
sudo docker compose up -d
```

### Backups

The critical data lives in the `stalwart-data` volume. Back it up
regularly:

```sh
# Stop stalwart to ensure consistency (optional but recommended)
sudo docker compose stop stalwart

# Backup
sudo tar czf /backup/stalwart-$(date +%Y%m%d).tar.gz -C /opt/mail stalwart-data/

# Restart
sudo docker compose start stalwart
```

For Caddy, back up `caddy-data/` to preserve TLS certificates (avoids
rate-limit issues with Let's Encrypt on restore).

### Migrating mail from another server

Use the `migrate-imap.sh` script from this repository:

```sh
SRC_HOST=old-mail.example.com DST_HOST=mail.example.com \
  ./scripts/migrate-imap.sh accounts.conf
```

See the [scripts README](README.md) for details.

## Troubleshooting

### Stalwart won't start

```sh
sudo docker compose logs stalwart
```

Common causes:
- Port 25 already in use (check `ss -tlnp | grep :25`)
- Insufficient permissions on `stalwart-data/`

### Caddy can't obtain certificates

- Verify ports 80 and 443 are reachable from the internet
- Verify DNS A/AAAA records point to this server
- Check `sudo docker compose logs caddy` for ACME errors

### Mail rejected by receiving servers

- Verify PTR record matches `mail.example.com`
- Verify SPF, DKIM, and DMARC records are published
- Run `./scripts/setup-dns.ts check` to verify all records
- Check your IP reputation at [multirbl.valli.org](https://multirbl.valli.org)

### TMail not loading

- Verify the `IMAP_SERVER`, `SMTP_SERVER` environment variables in the
  compose file point to the correct Stalwart container name
- Check `sudo docker compose logs tmail`

## Security notes

- Docker is running as root — Stalwart binds to privileged port 25.
  If you want to reduce the attack surface, consider using
  `cap_add: [NET_BIND_SERVICE]` instead of running the entire
  container as root, or use host-level port forwarding from 25 to
  an unprivileged port.
- Keep images updated. Stalwart and TMail publish regular security
  updates.
- The Stalwart admin UI is exposed on `https://mail.example.com` — use
  a strong password and consider restricting access by IP if possible.
- TLS certificates are managed by Caddy (for HTTPS) and by Stalwart
  (for SMTP/IMAP). Both handle renewal automatically.
